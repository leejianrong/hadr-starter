# Python 3.12 env: use `uv`, not the system `pip`

**Cost:** ~15 min of dead ends before switching to `uv`.

## Symptom

On this box, `python3` is 3.10, `python3.11` has a stray global `pytest`, and
`python3.12` (the project target) exists but its `pip` is broken:

```
ModuleNotFoundError: No module named 'distutils'
```

(distutils was removed in 3.12; the system pip shipped here predates that.) So
`python3.12 -m pip install ...` fails, and running tests under whichever
interpreter happened to have `pytest` risks version drift from the 3.12 target.

## Fix

Manage the project with `uv` + `pyproject.toml` (`requires-python = ">=3.12"`),
which provisions its own 3.12 and an isolated `.venv` with a lockfile:

```
uv sync            # creates .venv, installs deps + dev group from uv.lock
uv run pytest      # always the pinned 3.12 env
uv run python -m scripts.sitrep
```

## Lesson

Don't fight the system Python's packaging. Pin the interpreter and deps in
`pyproject.toml`/`uv.lock` and always go through `uv run` — reproducible, and it
sidesteps the distutils/pip breakage entirely. `.venv/` is git-ignored; `uv.lock`
is committed.
